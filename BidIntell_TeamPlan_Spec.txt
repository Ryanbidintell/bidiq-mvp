Strategic Context
Every specialty subcontracting firm is a company — not a collection of individuals. BidIntell currently signs up users one at a time, which means a shop with three estimators needs three separate accounts, three separate subscriptions, and zero shared visibility. That is backwards from how these businesses actually operate.

Shifting to company-first accounts fixes this at the foundation. The owner creates one company account, invites estimators, and manages everything from a single subscription. The tool becomes a company investment rather than a personal tool — which changes both the buying decision and the adoption dynamic.

WHY THIS ACCELERATES ADOPTION
When an owner buys BidIntell for their company, estimator adoption becomes a management decision rather than an individual choice. The outcome logging problem — the single most critical behavior for the data moat — is solved by accountability, not nudges.

RLS Policy Changes
Existing RLS policies use user_id = auth.uid(). Owner-role access requires a second policy layer — owners can read all rows where org_id matches their org.

CRITICAL RULE
Never drop existing RLS policies. Add new policies alongside them. A solo user with org_id = null must still be covered by user_id = auth.uid(). Both policies coexist.
projects table — new owner read policy
Add this policy alongside the existing user_id policy:

CREATE POLICY "Org owners can read all org projects"
ON projects FOR SELECT
USING (
  org_id IN (
    SELECT org_id FROM org_members
    WHERE user_id = auth.uid()
    AND role = 'owner'
  )
);

Apply the same pattern to user_settings and clients for owner-level reads.

DEFINITION OF DONE
Existing beta users can still log in, analyze bids, and view reports with no change to behavior. Zero regressions. Verified by manual test on at least 2 existing accounts.
Phase B — Company Signup & Invite Flow
New onboarding path for owners creating a company account.

Company Signup Flow
1. Owner goes to bidintell.ai — lands on updated pricing page showing all 4 tiers
2. Clicks "Start Free Trial" on any tier — redirects to auth.html with tier param (?tier=team)
3. Magic link auth — same as today
4. After auth: onboarding wizard detects tier param — adds new Step 0: "Set up your company"
5. Owner enters: Company Name, their name, their role (Owner / Principal / Estimator)
6. organizations row created — owner added to org_members with role = owner
7. Remaining onboarding steps proceed as normal (trade, location, GCs, etc.)
8. After onboarding: lands on dashboard — sees new "Team" tab (owner only)

Invite Flow (Owner sends invite)
9. Owner goes to Team tab ? clicks "Invite Estimator"
10. Modal: enter first name, last name, email address
11. Postmark sends invite email from hello@bidintell.ai — template: "You've been invited to join [Company] on BidIntell"
12. Email contains magic link with invite token (?invite=TOKEN)
13. Invitee clicks link ? auth.html ? magic link auth
14. After auth: abbreviated onboarding (name pre-filled, company pre-filled, just needs trade + location)
15. org_members row created — invite_accepted_at populated
16. Invitee lands on standard estimator dashboard — no Team tab

Seat limit enforcement: if org is at seat_limit, "Invite Estimator" button is disabled with tooltip: "Upgrade your plan to add more estimators."

Phase D — Billing Changes
Billing moves from per-user to per-organization. The Stripe customer and subscription belong to the org, not the individual.

• stripe-create-checkout.js reads org_id from session — creates Stripe customer for the org
• Checkout metadata includes org_id so webhook can write to the right organization row
• stripe-webhook.js updated to write subscription data to organizations table (not user_revenue for multi-seat plans)
• getSubscriptionStatus() checks organizations.subscription_status when user has an org_id
• Solo users (org_id = null) continue to use user_revenue — no change
• Stripe customer portal links are org-level — owner manages billing, estimators cannot

What This Unlocks

For Sales
• Every prospect call now has a clear question: "How many estimators does your team have?" — maps directly to a tier
• Owner buying for the team = management decision, not individual opt-in
• $99 Team plan is the natural first upgrade for any shop with 2+ estimators
• "Enterprise" tier signals credibility to larger shops without a self-serve ceiling

For the Data Moat
• Outcome logging accountability: owner can see who is logging and who is not
• Org-level win rate data compounds faster with multiple estimators feeding the system
• Trade win rate aggregation across estimators becomes meaningful at 50+ bids — a 3-person team gets there 3x faster

For the Exit Story
• Company accounts = enterprise software, not personal tool — changes valuation multiple framing
• Data keyed to organizations (not individuals) is more acquirer-friendly — survives employee turnover
• Estimator performance analytics is the exact data layer Procore and Autodesk would want to acquire
